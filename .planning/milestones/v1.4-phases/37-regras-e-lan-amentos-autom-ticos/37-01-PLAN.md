    **Step 3 — auto-posting.service.ts:**

    Create `apps/backend/src/modules/auto-posting/auto-posting.service.ts` with these exported functions:

    a) **`process(sourceType, sourceId, organizationId)`** — Main entry point called by hooks (per D-26):
       1. Check idempotency: query `pendingJournalPosting.findFirst({ where: { sourceType, sourceId, status: 'COMPLETED' } })`. If found, return silently (per D-17).
       2. Find active rule: `accountingRule.findFirst({ where: { organizationId, sourceType, isActive: true }, include: { lines: { include: { account: true }, orderBy: { lineOrder: 'asc' } } } })`. If none, return without creating pending (per D-18).
       3. Upsert PendingJournalPosting with status=PENDING (upsert by sourceType+sourceId to handle re-calls).
       4. Update status to PROCESSING.
       5. Call `_executePosting(pending, rule, organizationId)`.

    b) **`_executePosting(pending, rule, organizationId)`** — Private helper:
       1. Look up data extractor from `EXTRACTORS` map for `pending.sourceType`. Call it with `(pending.sourceId, organizationId)`.
       2. Resolve `periodId` from entryDate: `accountingPeriod.findFirst({ where: { organizationId, year: date.getFullYear(), month: date.getMonth() + 1, status: 'ABERTO' } })`. If not found or closed, update pending to ERROR "Periodo contabil fechado" (per D-25).
       3. Interpolate historyTemplate with templateVars using simple `template.replace(/\{\{(\w+)\}\}/g, (_, key) => vars[key] ?? '{{' + key + '}}')`.
       4. Infer costCenterId: if extractedData.farmId, query `costCenter.findFirst({ where: { organizationId, farmId } })`. If rule.requireCostCenter and no CC found, set ERROR.
       5. Create JournalEntry + Lines atomically in a Serializable transaction (replicating the posting logic from postJournalEntry but skipping allowManualEntry check per RESEARCH pitfall 1):
          - `prisma.journalEntry.create({ data: { organizationId, entryDate, periodId, description: interpolatedDescription, entryType: 'AUTOMATIC', status: 'POSTED', sourceType, sourceId, costCenterId, createdBy: 'system', entryNumber: nextNumber, postedAt: new Date(), lines: { create: rule.lines.map((rl, i) => ({ accountId: rl.accountId, side: rl.side, amount: extractedData.amounts[i].amount, description: rl.description ? interpolate(rl.description, templateVars) : null, lineOrder: rl.lineOrder, costCenterId })) } } })`
          - Run AccountBalance upsert for each line (same logic as postJournalEntry lines 256-301).
          - Run closingBalance recomputation raw SQL (same as postJournalEntry lines 290-301).
       6. Update PendingJournalPosting to COMPLETED with journalEntryId and processedAt.
       7. On any error: update to ERROR with errorMessage.

    c) **`retry(pendingId, organizationId)`** — Per D-16, resets existing pending to PENDING and re-executes (never creates new record per RESEARCH pitfall 2):
       1. Find pending by id+orgId. If not found or status not ERROR, throw.
       2. Update status to PENDING, clear errorMessage.
       3. Find active rule for pending.sourceType.
       4. Call `_executePosting(pending, rule, organizationId)`.

    d) **`retryBatch(filters, organizationId)`** — Retry all ERROR postings matching filters. Returns { succeeded: number, failed: number }.

    e) **`listPending(organizationId, filters)`** — Query PendingJournalPosting with optional status/sourceType filters, ordered by createdAt DESC.

    f) **`listRules(organizationId)`** — Query all AccountingRules for org with lines + account names, ordered by sourceType.

    g) **`getRule(organizationId, ruleId)`** — Single rule with lines.

    h) **`updateRule(organizationId, ruleId, input: UpdateRuleInput)`** — Per D-09 (future only). Update isActive, historyTemplate, requireCostCenter. If input.lines provided: delete existing lines, create new ones. Returns updated rule.

    i) **`previewRule(organizationId, ruleId)`** — Per D-03: find last PendingJournalPosting COMPLETED for this rule's sourceType, use its sourceId to extract data and build a preview entry. If no completed posting exists, return null.

    j) **`seedAccountingRules(organizationId)`** — Per D-02/D-13: creates default rules for all sourceTypes. Maps well-known COA codes from `coa-rural-template.ts` to accounts. Upserts by (organizationId, sourceType).

    k) **`EXTRACTORS: Record<AutoPostingSourceType, DataExtractor>`** — Map of data extractors (per D-28). Each extractor fetches data from the source module's table:
       - `PAYROLL_RUN_CLOSE`: fetch PayrollRun by id, aggregate totals from PayrollRunItem, return amounts for salary/charges/tax lines, templateVars: { referenceMonth, organizationName }.
       - `PAYABLE_SETTLEMENT`: fetch Payable by id with supplier, return settlement amount, templateVars: { supplierName, documentNumber, description }.
       - `RECEIVABLE_SETTLEMENT`: fetch Receivable by id, return amount, templateVars: { clientName, documentNumber, description }.
       - `DEPRECIATION_RUN`: fetch DepreciationRun by id, aggregate total depreciation, templateVars: { periodYear, periodMonth }.
       - `STOCK_ENTRY`: fetch StockEntry by id with items, return total cost, templateVars: { supplierName, documentNumber, referenceDate }.
       - `STOCK_OUTPUT_*`: fetch StockOutput by id with items, return total cost, templateVars: { outputType, referenceDate, productCount }.
       - `PAYROLL_PROVISION_*`: fetch PayrollProvision by id, return totalAmount, templateVars: { referenceMonth, provisionType }.
       - `PAYABLE_REVERSAL`/`RECEIVABLE_REVERSAL`: fetch original record, build reversal amounts (same amounts, swapped sides are handled by rule lines being reversed in the reversal rule).
