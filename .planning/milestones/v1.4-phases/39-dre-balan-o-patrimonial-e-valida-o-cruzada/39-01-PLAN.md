**dre.calculator.ts** — Pure function, NO Prisma imports. Uses Decimal.js for all arithmetic.

- `export function calculateDre(input: DreInput): DreOutput`
- Map accounts to sections using ACTUAL COA codes (per D-01 intent mapped to real template):
  - `4.1.xx` (excluding isFairValueAdj) → receita-bruta-agricola (receita agropecuaria includes pecuaria at this level, but split by looking at sub-accounts: 4.1.01 = grains/vegetal, 4.1.02 = pecuaria — use sub-code)
  - Actually, the template has 4.1 = "Receitas Agropecuarias" as synthetic. Its children: 4.1.01 = Venda Graos, 4.1.02 = Venda Animais, 4.1.03 = Variacao Valor Justo (isFairValueAdj), 4.1.04 = Receita Industrializacao. So: 4.1.01 → receita-bruta-agricola, 4.1.02 → receita-bruta-pecuaria, 4.1.04 → receita-bruta-industrializacao, any account with isFairValueAdj=true → cpc29 section.
  - `4.2.xx` → outras-receitas
  - `4.3.xx` → receita-financeira (treated as Other Income, not operational)
  - Deducoes: accounts whose code contains specific deduction codes. Per RESEARCH the template has no explicit deduction group. Use account name pattern matching or a `4.4.xx` group if present. If no deduction accounts exist yet, section shows zero. Actually check for FUNRURAL a Recolher at code `2.2.03.01` — this is a passivo account, not revenue deduction. For DRE deductions, look for any account under a deduction synthetic parent. If none exists, deducoes section = empty with zero total. The DRE calculator should gracefully handle empty sections.
  - `5.1.01.xx` → cpv-agricola
  - `5.1.02.xx` → cpv-pecuario
  - `5.2.01.xx` → despesas-admin
  - `5.2.02.xx` + `6.1.xx` → despesas-pessoal (merge legacy group 6 into pessoal)
  - `5.2.03.xx` → despesas-depreciacao
  - `5.3.xx` → despesas-financeiras
  - isFairValueAdj=true (any code) → cpc29

- For each account, compute net movement: for RECEITA (CREDORA), net = credit - debit (positive means revenue). For DESPESA (DEVEDORA), net = debit - credit (positive means expense). Display expenses as positive values in the DRE (section label communicates the sign).

- Compute subtotals:
  - Receita Bruta = sum(receita-bruta-agricola + receita-bruta-pecuaria + receita-bruta-industrializacao + receita-financeira + outras-receitas)
  - Receita Liquida = Receita Bruta - Deducoes
  - Lucro Bruto = Receita Liquida - CPV total
  - Total Despesas = sum(all despesas sections)
  - Resultado Antes IR = Lucro Bruto - Total Despesas +/- CPC 29
  - Resultado Liquido = Resultado Antes IR - IR/CSLL

- V/H analysis (always computed, frontend decides whether to show):
  - AV%: each row / Receita Liquida \* 100 (null if Receita Liquida = 0)
  - AH%: (currentMonth - priorYear) / abs(priorYear) \* 100 (null if priorYear = 0)

- Only sum ANALYTIC accounts (isSynthetic=false) to avoid double-counting.

  - ROE: resultadoLiquido / plTotal. Null if plTotal.isZero().
  - PL/ha: plTotal / totalAreaHa. Null if totalAreaHa = 0.
- All indicator values: Decimal.toFixed(2) for ratios, toFixed(2) for currency/ha.

**financial-statements.service.ts** — 3 exported async functions. Uses withRlsContext for multi-tenancy.

`getDre(organizationId: string, filters: DreFilters)`:

1. Load all active ChartOfAccount where accountType IN ('RECEITA', 'DESPESA') and isSynthetic=false.
2. Find current fiscal year by filters.fiscalYearId. Find prior fiscal year (previous year with same org).
3. **If NO costCenterId (consolidated):**
   - Query AccountBalance for current month: `where: { organizationId, fiscalYearId: filters.fiscalYearId, month: filters.month }`
   - Query AccountBalance for YTD: same fiscalYearId, months 1..filters.month, aggregate SUM(debitTotal), SUM(creditTotal) grouped by accountId
   - Query AccountBalance for prior year same month: use prior fiscalYearId + same month
4. **If costCenterId provided:**
   - Use raw SQL aggregation from JournalEntryLine JOIN JournalEntry:
     ```sql
     SELECT jel."accountId", jel.side, SUM(jel.amount) as total
     FROM journal_entry_lines jel
     JOIN journal_entries je ON je.id = jel."journalEntryId"
     WHERE je."organizationId" = $orgId
       AND jel."costCenterId" = $costCenterId
       AND je.status = 'POSTED'
       AND je."entryDate" BETWEEN $startDate AND $endDate
     GROUP BY jel."accountId", jel.side
     ```
   - Run 3 queries: current month dates, YTD dates, prior year same month dates.
5. Map query results to DreAccountData[].
6. Call `calculateDre(input)` and return result.
7. If costCenterId is NOT set, also compute MarginRankingItem[] by querying JournalEntryLine grouped by costCenterId for revenue (accounts 4.x) and CPV (accounts 5.1.x), computing gross margin per CC. Return top CCs sorted by margin descending.

`getBalanceSheet(organizationId: string, filters: BpFilters)`:

1. Load all active ChartOfAccount where accountType IN ('ATIVO', 'PASSIVO', 'PL') and isSynthetic=false.
2. Query AccountBalance for current month closingBalance.
3. Query AccountBalance for prior month closingBalance (month-1, or month 12 of prior fiscal year if month=1).
4. Get total farm area: `prisma.farm.aggregate({ where: { organizationId }, _sum: { totalAreaHa: true } })`.
5. Get resultadoLiquido from getDre (call getDre internally for the same period to get the net result for ROE).
6. For sparkline data: query AccountBalance for the last 6 months, compute each indicator per month, return as sparklines record.
7. Map to BpAccountData[], call `calculateBp(input)`, return result.

`getCrossValidation(organizationId: string, filters: { fiscalYearId: string; month: number })`:

1. Get DRE resultadoLiquido by calling getDre.
2. Get BP data by calling getBalanceSheet.
3. Compute deltaLucrosAcumulados: find "Lucros/Prejuizos Acumulados" account (code startsWith '3.3'), get closingBalance for current month minus closingBalance for prior month.
4. Get trial balance totals by calling ledger service getTrialBalance (import from ledger module).
5. Build CrossValidationInput and call `calculateCrossValidation(input)`.
