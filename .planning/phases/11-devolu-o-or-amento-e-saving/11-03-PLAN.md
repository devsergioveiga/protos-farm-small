**purchase-budgets.service.ts** — Create with these exported functions:

1. `createPurchaseBudget(ctx: RlsContext & { userId: string }, input: CreatePurchaseBudgetInput)`:
   - Validate periodEnd > periodStart — throw 400 "Data fim deve ser posterior a data inicio"
   - Validate no overlapping budget for same category+farm+period — throw 409 "Ja existe orcamento para esta categoria no periodo"
   - Create PurchaseBudget record
   - Return PurchaseBudgetOutput

2. `updatePurchaseBudget(ctx, id, input: UpdatePurchaseBudgetInput)`:
   - Find budget, validate exists
   - Update allowed fields (budgetedAmount, notes, periodEnd)
   - Return PurchaseBudgetOutput

3. `listPurchaseBudgets(ctx, query: ListPurchaseBudgetsQuery)`:
   - Paginated list with filters (farmId, category, periodType, date range)
   - Return ListPurchaseBudgetsResult

4. `deletePurchaseBudget(ctx, id)`:
   - Hard delete (budgets are planning records, not transactional)

5. `getBudgetExecution(ctx, query: { farmId?, periodStart?, periodEnd? })`:
   - For each PurchaseBudget matching filters, calculate in real-time:
     - **Requisitado**: SUM of PurchaseRequestItem.estimatedTotalPrice (or quantity \* estimatedUnitPrice) where RC status in ['APROVADA','EM_COTACAO','COMPRADA'] AND RC createdAt between budget periodStart/periodEnd. To match by category: join PurchaseRequestItem -> Product -> determine category. If no direct category field on PurchaseRequestItem, use the RC's type field to map to SupplierCategory (INSUMO_AGRICOLA for 'insumo_agricola', PECUARIO for 'pecuario', etc.) — the PurchaseRequest.type field likely maps to SupplierCategory.
     - **Comprado**: SUM of PurchaseOrderItem.totalPrice where OC status in ['EMITIDA','CONFIRMADA','EM_TRANSITO','ENTREGUE'] AND OC createdAt between periodStart/periodEnd. Category via same RC-item-to-budget mapping through purchaseRequestItemId.
     - **Pago**: SUM of Payable.totalAmount where Payable.status = 'PAGA' AND Payable.dueDate (or paidAt) between periodStart/periodEnd AND NOT isCredit. Category via Payable.category mapping to SupplierCategory.
   - Calculate percentUsed = (comprado / budgetedAmount) \* 100
   - Return BudgetExecutionResult with totals row

6. `getDeviationReport(ctx, query)`:
   - Filter getBudgetExecution rows where percentUsed > 100
   - For each over-budget category, include the RCs and OCs that contributed to the overrun
   - Return array of deviation entries

7. `checkBudgetExceeded(tx: TxClient, organizationId: string, category: string, farmId: string | null, amount: number)`:
   - Find active PurchaseBudget for this category + farm where now() is between periodStart and periodEnd
   - If no budget defined, return { exceeded: false }
   - Calculate current "comprado" total (same query as getBudgetExecution but for single category)
   - If currentSpent + amount > budgetedAmount, return { exceeded: true, budgetId, budgetedAmount, currentSpent, percentUsed }
   - Otherwise return { exceeded: false }

**purchase-budgets.routes.ts** — Create Express router under `/org/purchase-budgets`:

- `GET /` — listPurchaseBudgets (purchases:read)
- `GET /execution` — getBudgetExecution (purchases:read) — MUST be before /:id
- `GET /deviations` — getDeviationReport (purchases:read) — MUST be before /:id
- `POST /` — createPurchaseBudget (purchases:manage)
- `GET /:id` — getPurchaseBudget (purchases:read)
- `PUT /:id` — updatePurchaseBudget (purchases:manage)
- `DELETE /:id` — deletePurchaseBudget (purchases:manage)

**purchase-requests.service.ts** — Inject budget check into the approve function:

- After the status transition to APROVADA succeeds (inside the same transaction):
  - Determine the category from the RC's type or items
  - Call checkBudgetExceeded with the RC's estimated total
  - If exceeded, set `budgetExceeded = true` on the PurchaseRequest record
  - Include `budgetWarning` in the return value (or response metadata) — the service should return a `budgetWarning?: BudgetCheckResult` alongside the updated RC

**purchase-orders.service.ts** — Inject budget check into the transition function when transitioning to EMITIDA:

- After status update inside transaction:
  - Determine category from PO items (via purchaseRequestItemId chain or PO's RC reference)
  - Call checkBudgetExceeded with OC total
  - If exceeded, set `budgetExceeded = true` on PurchaseOrder record
  - Include budgetWarning in response
