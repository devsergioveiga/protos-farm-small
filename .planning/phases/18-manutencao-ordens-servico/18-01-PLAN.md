**maintenance-plans.service.ts:**

Implement the following functions following the assets.service.ts pattern (RLS context, Prisma queries):

1. `createMaintenancePlan(ctx: RlsContext, input: CreateMaintenancePlanInput)`:
   - Verify asset belongs to org via `prisma.asset.findFirst({ where: { id: input.assetId, organizationId: ctx.organizationId } })`
   - Call `computeNextDue()` to set initial nextDueAt/nextDueMeter
   - For CALENDAR: `nextDueAt = addDays(new Date(), input.intervalValue)` using date-fns
   - For HOURMETER/ODOMETER: query latest MeterReading for asset, set `nextDueMeter = lastMeterValue + intervalValue`
   - Create MaintenancePlan record with computed nextDue values
   - Return plan with asset include

2. `listMaintenancePlans(ctx: RlsContext, query: ListMaintenancePlansQuery)`:
   - Filter by organizationId (RLS), optional assetId, triggerType, isActive, farmId (via asset.farmId)
   - Include asset { id, name, assetTag }
   - Paginate with skip/take from query.page and query.limit (default 20)
   - Return { data, total, page, limit }

3. `getMaintenancePlan(ctx: RlsContext, id: string)`:
   - findFirst with org filter, include asset and workOrders (last 10, ordered by openedAt desc)
   - Throw MaintenancePlanError('Plano nao encontrado', 404) if null

4. `updateMaintenancePlan(ctx: RlsContext, id: string, input: UpdateMaintenancePlanInput)`:
   - Verify plan belongs to org
   - If triggerType or intervalValue changed, recalculate nextDue
   - Update record

5. `deleteMaintenancePlan(ctx: RlsContext, id: string)`:
   - Verify plan belongs to org
   - Delete record (hard delete — linked workOrders keep their maintenancePlanId as null via onSetNull or just leave FK)

6. `computeNextDue(triggerType, intervalValue, lastExecutedAt, lastMeterValue)`:
   - CALENDAR: `addDays(lastExecutedAt ?? new Date(), intervalValue)` → returns { nextDueAt, nextDueMeter: null }
   - HOURMETER/ODOMETER: `{ nextDueAt: null, nextDueMeter: (lastMeterValue ?? 0) + intervalValue }`
   - Export this function for reuse in work-orders close

7. `processOverduePlans(organizationId?: string)`:
   - Query all active plans where CALENDAR nextDueAt <= now OR HOURMETER/ODOMETER plans that need meter check
   - For each overdue plan: create notification via existing notifications.service.ts
   - Return count of overdue plans found

**maintenance-alerts.cron.ts:**
Follow exact pattern from digest.cron.ts:

```typescript
import cron from 'node-cron';
import { redis } from '../../database/redis';
import { processOverduePlans } from '../../modules/maintenance-plans/maintenance-plans.service';

export function startMaintenanceAlertsCron(): void {
  cron.schedule(
    '0 6 * * *', // Daily at 06:00 BRT
    async () => {
      const lockKey = `cron:maintenance-alerts:${new Date().toISOString().slice(0, 10)}`;
      const locked = await redis.set(lockKey, '1', 'EX', 3600, 'NX');
      if (!locked) return;
      try {
        await processOverduePlans();
      } finally {
        await redis.del(lockKey);
      }
    },
    { timezone: 'America/Sao_Paulo' },
  );
}
```

Register in app.ts or main.ts alongside other crons: `startMaintenanceAlertsCron()`.
