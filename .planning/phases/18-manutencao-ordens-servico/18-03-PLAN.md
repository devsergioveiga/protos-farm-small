**spare-parts.service.ts:**

Simple CRUD for SparePartAssetCompat join table:

1. `addSparePartCompat(ctx, input: { productId: string, assetId: string, notes?: string })`:
   - Verify asset belongs to org
   - Create SparePartAssetCompat (upsert to handle duplicates gracefully)
   - Return created record

2. `removeSparePartCompat(ctx, id: string)`:
   - Verify compat record exists for org (via asset.organizationId)
   - Delete record

3. `listSparePartsForAsset(ctx, assetId: string)`:
   - Query SparePartAssetCompat where assetId, join Product for name/unit
   - Also join StockBalance for current stock level of each product
   - Return array of { productId, productName, unit, currentStock, reorderPoint, notes }

4. `listAssetsForSparePart(ctx, productId: string)`:
   - Query SparePartAssetCompat where productId, join Asset for name/assetTag
   - Return array of { assetId, assetName, assetTag }

**spare-parts.routes.ts:**

- `POST /` — authenticate, authorize('spare-parts:create'), addSparePartCompat
- `DELETE /:id` — authenticate, authorize('spare-parts:update'), removeSparePartCompat
- `GET /by-asset/:assetId` — authenticate, authorize('spare-parts:read'), listSparePartsForAsset
- `GET /by-product/:productId` — authenticate, authorize('spare-parts:read'), listAssetsForSparePart

Register: `app.use('/api/org/spare-parts', sparePartsRouter)`

**maintenance-provisions.service.ts:**

1. `createProvision(ctx, input: CreateMaintenanceProvisionInput)`:
   - If assetId provided, verify asset belongs to org
   - Create MaintenanceProvision record
   - Return with asset include

2. `listProvisions(ctx, query: ListMaintenanceProvisionsQuery)`:
   - Filter by org, optional assetId, isActive
   - Include asset { id, name, assetTag }
   - Paginate

3. `updateProvision(ctx, id: string, input: UpdateMaintenanceProvisionInput)`:
   - Verify belongs to org, update

4. `deleteProvision(ctx, id: string)`:
   - Verify belongs to org, delete

5. `getReconciliation(ctx, query: { year: number, month: number })`:
   - **Provisioned**: sum monthlyAmount for active provisions in org
   - **Actual**: sum totalCost from WorkOrders closed in that year/month (use closedAt month/year per RESEARCH.md Pitfall 7)
   - **Per-asset breakdown**: for each asset with either provision or actual cost, compute variance
   - Return ProvisionReconciliationOutput

6. `processMonthlyProvisions()`:
   - Query all active provisions across all orgs
   - For each: log the provision entry (or integrate with financial module if CP entries exist)
   - This is the monthly automation hook — for Phase 18, it simply marks "provision processed for period" in a log. Full financial integration deferred to Phase 23.

**maintenance-provisions.routes.ts:**

- `POST /` — authenticate, authorize('maintenance-provisions:create'), createProvision
- `GET /` — authenticate, authorize('maintenance-provisions:read'), listProvisions
- `GET /reconciliation` — authenticate, authorize('maintenance-provisions:read'), getReconciliation (BEFORE /:id)
- `PUT /:id` — authenticate, authorize('maintenance-provisions:update'), updateProvision
- `DELETE /:id` — authenticate, authorize('maintenance-provisions:update'), deleteProvision

Register: `app.use('/api/org/maintenance-provisions', maintenanceProvisionsRouter)`

**maintenance-provision.cron.ts:**
Follow digest.cron.ts pattern:

- Schedule: `'0 2 1 * *'` (1st of month at 02:00 BRT)
- Redis lock key: `cron:maintenance-provision:${year}-${month}`
- Call `processMonthlyProvisions()`
- Register in app.ts/main.ts alongside other crons
