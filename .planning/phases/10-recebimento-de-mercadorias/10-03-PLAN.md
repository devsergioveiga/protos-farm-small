Implementation inside withRlsContext(ctx, async (tx) => { ... }):

**Step 1 — Load and validate:**

- Load GR with items, divergences, purchaseOrder (include items, quotation with purchaseRequest and QuotationProposal with paymentTerms), supplier
- Throw GoodsReceiptError 404 if not found
- Throw GoodsReceiptError 400 if !canGrTransition(gr.status, 'CONFIRMADO')
- For STANDARD/PARCIAL/NF_FRACIONADA: require invoiceNumber and invoiceTotal (cannot confirm without NF)
- For MERCADORIA_ANTECIPADA: invoiceNumber NOT required (that's the point — goods before NF)
- For NF_ANTECIPADA: goods must have arrived (check receivedAt is set)

**Step 2 — Derive farmId and costCenter:**

- farmId from PurchaseRequest (via PO -> Quotation -> PurchaseRequest.farmId); fall back to storageFarmId
- If EMERGENCIAL (no PO): use storageFarmId (required for EMERGENCIAL)
- costCenterId from PurchaseRequest.costCenterId (if available)

**Step 3 — Build and create StockEntry (skip for NF_ANTECIPADA if goods not yet received):**

- Map GR items to StockEntryItemInput: { productId: item.productId!, quantity: Number(item.receivedQty), unitCost: Number(item.unitPrice), batchNumber: item.batchNumber, expirationDate: item.expirationDate?.toISOString() }
- For items where productId is null (free-text emergency items): skip stock entry for those items
- Set initialStatus = 'DRAFT' if receivingType === 'MERCADORIA_ANTECIPADA', else 'CONFIRMED'
- Set goodsReceiptId = gr.id
- Call createStockEntry(ctx, stockInput, ctx.userId) — NOTE: since createStockEntry internally calls withRlsContext which starts its own transaction, and we need atomicity, we must INLINE the stock entry creation logic using the tx client directly. Copy the essential create logic from createStockEntry but use `tx` instead of creating a new transaction. Alternatively, extract the inner logic of createStockEntry into a `createStockEntryTx(tx, orgId, input, userId)` function.

**IMPORTANT DECISION on transaction nesting:** Since both createStockEntry and createPayable use withRlsContext internally (which starts its own transaction), we CANNOT call them directly inside our withRlsContext. Two approaches:

- **Approach A (recommended):** Inline the essential Prisma creates for StockEntry and Payable directly using the tx client. This avoids nested transaction issues. Create StockEntry record, StockEntryItems, update StockBalance — replicate the core logic.
- **Approach B:** Refactor createStockEntry/createPayable to accept an optional tx client parameter. This is cleaner but modifies shared code.

Use Approach A — inline the creates using tx. The key operations are:

1. `tx.stockEntry.create(...)` with items nested
2. For each item with productId: `tx.stockBalance.upsert(...)` to update balance (increment quantity, recalc average cost)
3. `tx.payable.create(...)` with installments nested

**Step 4 — Build and create Payable:**

- Parse paymentTerms from QuotationProposal (e.g., "30/60/90" -> installmentCount=3, first due = invoiceDate + 30 days)
- If no paymentTerms or "A vista": installmentCount=1, dueDate = invoiceDate + 30 days (default)
- supplierName from gr.supplier.name
- category: 'INPUTS' (PayableCategory enum)
- description: `NF ${gr.invoiceNumber} - ${gr.sequentialNumber}`
- totalAmount: Number(gr.invoiceTotal)
- documentNumber: gr.invoiceNumber
- farmId: derived in Step 2
- costCenterItems: [{ costCenterId, farmId, allocMode: 'PERCENTAGE', percentage: 100 }]
- Set originType: 'GOODS_RECEIPT', originId: gr.id, goodsReceiptId: gr.id
- For NF_ANTECIPADA: create Payable but set isProvisional on GR (Payable status is normal PENDING)
- For MERCADORIA_ANTECIPADA: do NOT create Payable yet (no NF = no financial obligation)
- Create Payable with installments (use generateInstallments if available, or create manually)

**Step 5 — Update PO receivedQuantity (partial delivery tracking):**

- For each GR item with purchaseOrderItemId: use Prisma increment:
