  <action>
    Create `modules/payables/` directory with colocalized module files.

    **payables.types.ts:**
    - PayableError class (extends Error with statusCode, like BankAccountError)
    - CreatePayableInput: { farmId, producerId?, supplierName, category (PayableCategory), description, totalAmount (number), dueDate (string ISO), documentNumber?, installmentCount (default 1), costCenterItems: CostCenterItemInput[], recurrenceFrequency?, recurrenceEndDate?, bankAccountId?, notes? }
    - UpdatePayableInput: partial of Create (only PENDING payables editable)
    - SettlePaymentInput: { paidAt (string ISO), amount (number), bankAccountId, interestAmount?, fineAmount?, discountAmount? }
    - BatchSettleInput: { bankAccountId, paidAt, items: { payableId, amount, interestAmount?, fineAmount?, discountAmount? }[] }
    - PayableOutput, PayableInstallmentOutput, PayableCostCenterItemOutput interfaces
    - ListPayablesQuery: { farmId?, status?, category?, startDate?, endDate?, search?, page?, limit? }

    **payables.service.ts:**
    Follow bank-accounts.service.ts pattern (withRlsContext, handleError).

    1. `createPayable(ctx, input)` — validate costCenterItems via validateCostCenterItems(Money(totalAmount), items). Generate installments via generateInstallments(). Prisma transaction: create Payable + PayableInstallment[] + PayableCostCenterItem[]. If recurrenceFrequency set, mark this as recurrence template.

    2. `listPayables(ctx, query)` — paginated list with filters (farmId, status, category, date range, search on supplierName/description/documentNumber). Include installments and costCenterItems. Order by dueDate ASC.

    3. `getPayable(ctx, id)` — single payable with all relations.

    4. `updatePayable(ctx, id, input)` — only PENDING status. If installmentCount or totalAmount changes, regenerate installments. Revalidate costCenterItems.

    5. `deletePayable(ctx, id)` — only PENDING. Soft delete via CANCELLED status (not hard delete).

    6. `settlePayment(ctx, payableId, input)` — atomic Prisma transaction per Research Pattern 2: (a) update Payable status to PAID with settlement fields, (b) decrement BankAccountBalance.currentBalance by effectiveAmount (amount + interest + fine - discount), (c) create FinancialTransaction with type='DEBIT', referenceType='PAYABLE'. Use Money for all arithmetic.

    7. `batchSettlePayments(ctx, input)` — single Prisma transaction wrapping multiple settle operations. All succeed or all fail (Pitfall 6 from research). Returns array of results.

    8. `reversePayment(ctx, payableId)` — only PAID payables. Atomic transaction: set status back to PENDING, clear settlement fields, increment BankAccountBalance by amountPaid, create FinancialTransaction with type='CREDIT' and description "Estorno CP #...".

    9. `generateRecurrence(ctx)` — find all payables with recurrenceFrequency set, no recurrenceEndDate or endDate in future, check if next occurrence should be generated (based on last child dueDate + frequency). Create new Payable as child (recurrenceParentId = template.id).

    **payables.routes.ts:**
    Express Router under `/org/payables`. Use `checkPermission('financial', 'read'|'create'|'update'|'delete')`.
    - GET `/` — listPayables
    - GET `/:id` — getPayable
    - POST `/` — createPayable
    - PUT `/:id` — updatePayable
    - DELETE `/:id` — deletePayable
    - POST `/:id/settle` — settlePayment
    - POST `/batch-settle` — batchSettlePayments (register BEFORE /:id routes)
    - POST `/:id/reverse` — reversePayment
    - POST `/generate-recurrence` — generateRecurrence

    **app.ts:** Import and register payablesRouter.

    **payables.routes.spec.ts:**
    Jest tests covering: create with installments (verify first-installment residual), create with rateio (percentage + fixed_value), list with filters, settle with juros/multa/desconto (verify bank balance decremented), batch settle, reverse (verify bank balance restored), recurrence generation. Follow bank-accounts.routes.spec.ts pattern for test setup.
